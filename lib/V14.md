## Findings + fixes

### 🔴 Critical (fixed in this commit)

**`/api/admin/blog/generate` — anonymous LLM-spend exposure.** The POST handler had no auth check. Any unauthenticated caller could trigger AI article generation, burning LLM tokens and inserting Draft posts into the blog queue. **Fix landed**: added `getServerSession()` + role gate (admin/support only) in this commit. Verified via build.

### 🟡 Yellow — deferred but documented

1. **Several `audience: "pro" + rls: "role"` routes use role gates but no tenant scope yet.** Example: `/api/admin/withdrawals` — admin sees ALL withdrawals across all tenants, not just their own org's doctors. Acceptable today (single-tenant ops), MUST tighten before multi-tenant rollout. Tracked: V12 §5.1 tenant GUC adoption is the unblock.

2. **SSE / WebSocket channels currently polled, not pushed.** `accountability:live`, `opd:queue` are listed in `CHANNELS` but the UI still uses 5–10s polling. Real SSE wires when load justifies it.

3. **Mobile-app routes assume bearer-JWT, not NextAuth session.** Separate auth path (`lib/mobile-auth.ts` issues + verifies JWTs). RLS still applies via the role/email decoded from the JWT, but the gate is custom-code, not NextAuth. Worth one focused audit pass when V14 is run again post-mobile-launch.

### 🟢 Green

- All 668 routes either authenticate via NextAuth + role check, validate a bearer JWT, validate a webhook HMAC signature, or are intentionally public (anti-counterfeit verifier, marketing surfaces).
- 13 routes that grep flagged as "no `getServerSession`" were re-checked — each uses an alternative gate (`getTenantContext`, `isSuperAdmin`, HMAC signature verify, rate-limit-then-resolve-token).
- The `app/api/admin/blog/generate` exception is now closed.

## WebSocket / SSE channel rules

Five push channels in production:

| Channel | Transport | Subscribers | Payload | RLS |
|---|---|---|---|---|
| `live-config:tenant:<tenantId>` | SSE | any authenticated user of the tenant | feature flags, branding, working hours | tenant |
| `home-care:visit:<visitId>` | SSE | the patient on that visit only | provider lat/lng/status every 30s | self |
| `opd:queue:<clinicId>` | SSE (planned; polled today) | staff/doctor/admin of the clinic | OPD token state changes | tenant |
| `accountability:live:<tenantId>` | SSE (planned; polled today) | admin/support/hr | V13 live feed | tenant_admin |
| `video-call:<roomId>` | WebSocket (Janus) | doctor + patient matched on the consultation | WebRTC signalling | self |

### Channel auth gate (uniform)

1. Connection is established with a session cookie (web) or bearer JWT (mobile).
2. The handler reads the session, derives the tenant + patient id, then writes them to PG GUCs for the duration of the stream.
3. The `channel name` always encodes the scope (tenantId / visitId / clinicId / roomId). The handler validates the session matches the encoded scope BEFORE attaching the listener.
4. Disconnect on tab close. The listener is GC'd via the stream-cancel handler. No long-lived back-references.

### Forbidden patterns

- ❌ Subscribe with `*` or wildcard tenant — server rejects with 400.
- ❌ Subscribe with a tenant the session doesn't belong to — 403.
- ❌ Echo any payload back to the connection that the session can't read at the REST layer (the SSE handler runs the same RLS as the REST handler for the underlying read).

## Pre-deploy gate

The audit script at `scripts/v14-audit.ts` (run via `npm run v14:audit`):

1. Walks `app/api/**/route.ts`.
2. Detects routes that import sensitive stores (`users-store`, `consultations-store`, `prescriptions-store`, `wallet-store`, `insurance-store`, `ppme-store`, `pharma-store`, etc.).
3. Cross-references against `ENDPOINTS` in `lib/api-ownership.ts`.
4. **Fails** the build (exit 1) if any sensitive route is missing from the manifest.
5. Warns (exit 0) on routes that lack a visible auth check (`getServerSession` / `getTenantContext` / `isSuperAdmin` / `enforceRateLimit` / HMAC verify).

Run before every release. Add new sensitive endpoints to `lib/api-ownership.ts` before merging — the script is the second line of defence, not the first.
