# V12 Database Audit — 47 Tables Added

V12 of the Master Spec resolved 47 missing tables, 10 cross-connection
gaps, and 14 naming inconsistencies in the platform schema. This doc is
the index — where each table lives, what it's for, and which V4–V13
surface it backs.

## Run order

```bash
# 1. Generate the CREATE TABLE migration from the schema
npm run db:generate

# 2. Apply tables to Postgres (Drizzle migrator)
npm run db:migrate

# 3. Apply RLS policies + TimescaleDB hypertables + composite indexes
#    that drizzle-kit can't express
npm run db:v12:policies
```

## V12 §6.1 — TimescaleDB hypertables

| Table | Reason | Chunk interval | Compression after |
|---|---|---|---|
| `icu_vitals`             | ~1 row/sec/bed | 1 day  | 7 days |
| `observation_vitals`     | Interval-driven, lower vol | 7 days | 30 days |
| `mar_administrations`    | Per-shift queries | 7 days | 30 days |
| `accountability_events`  | V13 live feed | 30 days | 90 days |

Compression is automatic via `add_compression_policy()`. Saves ~80% on
vitals after the working set ages out.

## V12 §6.2 — composite + partial indexes

Drizzle expresses simple composite indexes via the `index()` helper. The
following partial / GIN / trigram indexes are added via the SQL script
because drizzle-kit doesn't have helpers for them:

- `patients_name_trgm_idx` — GIN trigram on `patients.name` for the
  "search by name" box everywhere
- `doctors_name_trgm_idx` — same on `doctors.name`
- `equipment_products_title_trgm_idx` — same on the marketplace search
- `cars_open_by_deadline_idx` — partial index on
  `corrective_action_requests` WHERE status NOT IN closed/verified
- `accountability_unack_breaches_idx` — partial on events with an
  unacknowledged breach (the live-feed default filter)
- `lab_results_critical_unack_idx` + `lab_results_critical_only_idx` —
  V13 §3 30-min critical-value rule
- `wallet_txns_ref_compound_idx` — reconciliation by refKind + refId +
  createdAt

## V12 §5.1 — Row-Level Security

Three session-local GUCs the application sets on every request:

| GUC | Set by | Purpose |
|---|---|---|
| `odudoc.tenant_id`        | NextAuth middleware on every authenticated request | Hospital scoping. `'*'` = super-admin sees all |
| `odudoc.patient_id`       | Patient app + family-switcher | Patient-self read of own records |
| `odudoc.wallet_owner_id`  | Anyone who can have a wallet | Wallet + CAR + accountability self-visibility |

### Tenant-scoped tables (20)

`patients`, `encounters`, `admissions`, `discharges`, `prescriptions`,
`lab_orders`, `lab_results`, `icu_vitals`, `observation_vitals`,
`mar_administrations`, `departments`, `wards`, `beds`,
`ppme_reports`, `ppme_test_results`, `insurer_empanelments`,
`pre_auth_requests`, `insurance_claims`, `claim_documents`,
`near_miss_reports`.

Policy on each: `current_tenant() = '*' OR tenant_id = current_tenant() OR tenant_id IS NULL`.

### Patient-self read

`encounters`, `prescriptions`, `lab_orders`, `lab_results`,
`admissions`, `discharges` get a second SELECT policy that lets the
patient read their own rows even when `tenant_id` doesn't match
(because patient app calls don't carry a tenant context).

### Wallet visibility

`wallets`, `wallet_txns` — RLS by `entity_id = current_wallet_owner()`.

### Accountability + CARs

`accountability_events` — tenant scoped + the actor always sees their
own events (for the self-scorecard view).

`corrective_action_requests` — tenant + assignee + opener.

### Family graph

`family_members` — only patients in the row can see it.

## What's intentionally NOT in this commit

- **Backfill scripts** from the existing `bindPersistentArray` blobs to
  these tables. Each `lib/*-store.ts` currently writes to `app_kv`; the
  cutover script runs row-by-row from blob to table. Per-store
  cutover commits land separately so each can be reviewed + tested in
  isolation.
- **Code calls to set the RLS GUCs**. NextAuth callback needs an
  `await client.unsafe("SET LOCAL odudoc.tenant_id = '...'")` per
  request. Wiring lives in `lib/db.ts` (not this commit).
- **Hospital-tenant assignment on existing rows**. Today every store
  has implicit "platform" scope. Migrating rows to a real `tenant_id`
  is a one-shot UPDATE that depends on how each tenant rolls out.
- **PHI encryption-at-rest** (V12 §5.2). pgcrypto extension is created
  here so the column-level encryption path is unblocked; the actual
  application-side encryption helpers ship separately when the KMS
  integration lands.
- **Continuous aggregates** (TimescaleDB feature for materialised
  vital-sign roll-ups). Worth doing once we have real ICU data —
  premature otherwise.
